# 采用 shadcn/ui 默认颜色系统

## 概述

移除当前应用中的自定义颜色定义，完全采用 shadcn/ui 的默认颜色系统，以实现设计一致性并简化维护。

## 背景

Hagicode Desktop 应用使用 shadcn/ui 作为基础 UI 组件库，项目已集成 Tailwind CSS 4.0.0 和完整的 shadcn/ui 组件体系。当前存在自定义颜色配置，与 shadcn/ui 默认设计系统不一致。

## 问题

- **设计不一致**：自定义颜色与 shadcn/ui 默认设计系统不匹配
- **维护负担**：需要额外维护自定义颜色配置
- **组件样式冲突**：自定义颜色可能与 shadcn/ui 组件默认样式冲突
- **主题切换困难**：未来支持深色模式时，自定义颜色会增加实现难度

## 解决方案

### 技术方案

1. 移除 Tailwind 配置中的自定义颜色定义
2. 采用 shadcn/ui 默认设计令牌（CSS 变量）
3. 更新组件中的硬编码颜色值
4. 验证 UI 一致性和可访问性

### 实施策略

渐进式迁移，确保每个步骤都可以独立验证：
1. 先清理配置文件
2. 再更新组件样式
3. 最后全面验证

## 范围

### 包含

- 清理 `tailwind.config.js` 中的自定义颜色定义
- 更新全局 CSS 文件中的颜色变量
- 更新所有使用自定义颜色的组件
- 验证所有页面的视觉效果

### 排除

- shadcn/ui 组件的内部实现（由库管理）
- 非颜色相关的样式调整
- 主题切换功能（未来工作）

## 影响

### 预期收益

| 收益 | 描述 |
|------|------|
| 设计一致性 | UI 完全符合 shadcn/ui 设计规范 |
| 简化维护 | 移除自定义配置，降低维护成本 |
| 更好的主题支持 | 为深色模式等功能奠定基础 |
| 组件兼容性 | 确保 shadcn/ui 组件以预期方式渲染 |

### 风险评估

| 风险 | 缓解措施 |
|------|---------|
| 视觉变化 | 变更前截图对比，团队评审确认 |
| 回归问题 | 全面测试所有页面和组件 |
| 用户适应 | 变更说明，强调设计系统标准化 |

### 依赖

- 无外部依赖

### 受影响组件

- Tailwind 配置
- 全局 CSS 文件
- 所有 UI 组件（使用自定义颜色的）

## 成功标准

- [ ] 移除所有自定义颜色定义
- [ ] 所有组件使用 shadcn/ui 标准颜色类或变量
- [ ] 应用运行无样式错误
- [ ] 所有页面视觉效果一致且可访问
- [ ] 通过完整的 UI 回归测试
